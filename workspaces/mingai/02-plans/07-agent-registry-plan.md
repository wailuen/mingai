# 07 — Hosted Agent Registry: Implementation Plan

**Date**: 2026-03-05
**Duration**: 28 weeks (7 months) for Phase 1 + 2; Phase 3+ gated on traction metrics
**Team assumption**: 4-6 engineers + 1 product manager

---

## Strategic Framing

HAR is a Phase 4+ initiative for mingai — it should not be built at the expense of the core platform (tenant admin console, platform admin console). The correct sequencing is:

1. **Phase 0-1** (Months 1-6): Minimal viable registry alongside core platform. Build just enough to learn: will mingai tenants actually publish external agents? Do real transactions happen?
2. **Gate check**: 100+ real transactions in 6 months → proceed to Phase 2. Less than 100 → reassess concept before further investment.
3. **Phase 2+** (Months 7+): Full blockchain integration, open registry, financial settlement — only if traction is proven.

**Critical sequencing rule**: Do NOT build blockchain infrastructure before Phase 2. The first version is a signed audit log with traditional database — functionally equivalent for MVP, 80% less complexity.

## Phase 0: Foundation Without Blockchain (Weeks 1-6)

**Goal**: Validate the core concept with real mingai tenants. Can we get 5-10 agent card registrations and at least 10 real B2B transactions?

**Scope constraints**: No blockchain. No KYB. No third-party agents. Only mingai tenants. No financial transactions.

### Sprint 0-A (Weeks 1-3): Agent Card Registry

**Backend**:

```
POST /api/v1/registry/agents         — register agent card
GET  /api/v1/registry/agents         — list/search agents
GET  /api/v1/registry/agents/:id     — get agent card
PUT  /api/v1/registry/agents/:id     — update agent card
DELETE /api/v1/registry/agents/:id   — deregister agent

Agent Card Schema (v0.1):
{
  agent_id: string (UUID)
  tenant_id: string (FK to tenants table)
  name: string
  description: string
  transaction_types: string[]
  industries: string[]
  languages: string[]
  a2a_endpoint: string (HTTPS only)
  health_check_url: string
  status: "pending_kyb" | "active" | "suspended"
  trust_score: int (20-100)
  created_at: timestamp
}
```

**Storage**: PostgreSQL (new `agent_cards` table via Alembic migration; tenant_id FK to `tenants` table; RLS enabled)
**Search**: Azure AI Search index on agent cards (reuse existing search infrastructure)
**Health monitor**: Background job pings `health_check_url` every 5 minutes; marks agent UNAVAILABLE if 3 consecutive failures

### Sprint 0-B (Weeks 4-6): Registry UI + Tenant Integration

**UI** (Next.js, Shadcn):

- Workspace settings → Registry tab → "Publish to Global Registry" button
- Agent selection from workspace agents
- Agent card configuration form (transaction types, industries, languages, SLA)
- Registry search page (public-facing, no auth required to browse)
- Agent card detail page (trust score, transaction count, attestations)

**Tenant Admin integration**:

- Published agents shown with [Public] badge in workspace agent list
- Registry analytics widget on workspace dashboard: "Your agent was discovered X times this week"

**Deliverables**:

- Agent registration, update, deregister
- Basic discovery search (no AI ranking yet; sorted by registration date)
- Health monitoring
- 5 pilot tenant admins invited to publish

**Acceptance criteria**:

- At least 3 pilot tenants successfully publish an agent card
- Search returns correct results for industry + transaction type filters
- Health monitor correctly flags unhealthy endpoints

## Phase 1: A2A Protocol + Transaction Logging (Weeks 7-16)

**Goal**: Enable real A2A transactions between registered agents with signed audit log. Prove that autonomous B2B transactions can happen without blockchain complexity.

**Gate from Phase 0**: At least 3 registered agents and 1 attempted transaction.

### Sprint 1-A (Weeks 7-9): A2A Protocol Implementation

**A2A Message Structure**:

```json
{
  "header": {
    "message_id": "uuid",
    "from_agent_id": "did:mingai:acme:procurement-v1",
    "to_agent_id": "did:mingai:baker:sales-v1",
    "txn_id": "HAR-YYYYMMDD-NNNNNN",
    "message_type": "RFQ | QUOTE | PO | ACK | ...",
    "timestamp": "ISO-8601",
    "nonce": "random-32-bytes-hex"
  },
  "payload": {
    // message-type-specific structured data
    // JSON Schema enforced per message_type
  },
  "signature": "Ed25519 signature over SHA-256(header||payload)"
}
```

**Agent Key Management**:

- Each registered agent gets an Ed25519 keypair generated by HAR
- Private key stored in Azure Key Vault (per-agent secret)
- Public key in agent card (visible to all)
- Signature generation: HAR signs outbound messages on agent's behalf (Phase 1)
- Future (Phase 2): agents hold their own keys via BYOK

**HAR as message broker (Phase 1 simplification)**:

- Buyer's system calls HAR API → HAR signs and routes to Seller's A2A endpoint
- Seller's A2A endpoint responds → HAR records, forwards to Buyer
- This avoids requiring external developers to implement signing in Phase 1

### Sprint 1-B (Weeks 10-12): Transaction State Machine + Audit Log

**Transaction State Machine**:

```
DRAFT → OPEN → NEGOTIATING → COMMITTED → EXECUTING → COMPLETED
                              ↓                          ↓
                          ABANDONED                  DISPUTED → RESOLVED
```

**Audit Log** (traditional signed log, not blockchain):

```
CREATE TABLE har_transaction_events (
  id UUID PRIMARY KEY,
  txn_id VARCHAR NOT NULL,
  event_type VARCHAR NOT NULL,         -- RFQ_SENT, QUOTE_RECEIVED, PO_PLACED, etc.
  from_agent_id VARCHAR NOT NULL,
  to_agent_id VARCHAR NOT NULL,
  payload_hash VARCHAR NOT NULL,       -- SHA-256 of the event payload
  payload_encrypted TEXT,              -- encrypted payload (off ledger)
  platform_signature VARCHAR NOT NULL, -- HAR's Ed25519 signature over the event
  block_height INT,                    -- NULL in Phase 1; populated in Phase 2 (blockchain)
  timestamp TIMESTAMPTZ NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

**Signature chaining**: Each event's signature covers the previous event's hash + current event data. This creates a tamper-evident chain without blockchain. If any event is altered, all subsequent signatures become invalid.

**Phase 1 audit log properties**:

- Tamper-evident (signature chaining)
- Centrally controlled by mingai (NOT neutral — this is the limitation Phase 2 fixes)
- Sufficient for: internal audit, debugging, dispute evidence
- NOT sufficient for: external legal disputes where mingai might be conflicted

### Sprint 1-C (Weeks 13-16): Transaction Types + Human Approval

**Implemented in Phase 1** (Tier 1-2):

- CAPABILITY_QUERY (T1)
- RFQ / QUOTE_RESPONSE (T1)
- PO_PLACEMENT / PO_ACKNOWLEDGEMENT (T2)
- DELIVERY_CONFIRMATION (T4)

**Human Approval Gate**:

- Per-tenant configurable threshold: default $5,000, max $1,000,000
- Threshold per transaction type (e.g., RFQ: no approval; PO: $5,000+; financial: always)
- Approval via email link or in-app notification
- Timeout: 48 hours (configurable); expired approvals auto-reject

**Fee Recording** (not collected yet in Phase 1 — track only):

```
CREATE TABLE har_fee_records (
  id UUID PRIMARY KEY,
  txn_id VARCHAR NOT NULL,
  fee_type VARCHAR NOT NULL,      -- TIER_1, TIER_2, TIER_3
  amount_usd DECIMAL(10,4),
  status VARCHAR DEFAULT 'accrued', -- accrued | invoiced | paid
  created_at TIMESTAMPTZ
);
```

**Deliverables**:

- Full RFQ → PO flow working between registered agents
- Tamper-evident signed audit log
- Human approval gates at configurable thresholds
- Basic reconciliation: transaction record exportable as CSV

**Acceptance criteria**:

- At least 10 real transactions complete the RFQ → PO flow
- Human approval gate correctly blocks POs above threshold
- Audit log signature verification passes independently

**Phase 1 gate check** (end of week 16):

- 100+ transactions attempted? → Proceed to Phase 2
- <100 transactions → Stakeholder review: is the concept validated? Is the friction in the UX or the concept?

## Phase 2: Blockchain Integration + KYB (Weeks 17-24)

**Gate**: 100+ real transactions from Phase 1, stakeholder approval

**Goal**: Replace signed audit log with Hyperledger Fabric (permissioned blockchain). Add KYB verification. Enable Tier 3 (financial) transactions with escrow.

### Sprint 2-A (Weeks 17-19): Hyperledger Fabric Deployment

**Infrastructure**:

- 3-node Fabric network (1 orderer, 2 peers) on Azure Kubernetes Service
- 1 Fabric channel per transaction pair (data isolation by design)
- Chaincode (Go): TransactionContract, AgentRegistryContract
- Bridge: HAR API writes to Fabric on each state transition

**Migration from Phase 1**:

- Historical Phase 1 audit log remains in traditional database
- New transactions from Phase 2 onwards: on Fabric
- No migration of Phase 1 records to Fabric (not necessary; tamper-evidence properties preserved in Phase 1 log)

**Polygon CDK checkpoint layer** (simplified):

- Every 100 Fabric blocks: Fabric block hash → Polygon CDK checkpoint transaction
- Public verifiability: anyone can verify a transaction hash exists on a specific block

### Sprint 2-B (Weeks 20-21): KYB Integration

**Stripe Identity integration**:

- Tenant triggers KYB from agent registration flow
- Stripe Identity webhook: `identity.verification_session.verified` → HAR marks trust Level 1
- D&B Hoovers lookup: company registration cross-reference (for higher-trust tiers)

**Trust Score System**:

```python
def compute_trust_score(agent_id):
    base_attestation = get_kyb_level(agent_id) * 25  # 0, 25, 50, 75, 100
    txn_volume_score = min(log10(get_txn_count(agent_id) + 1) / log10(1000), 1.0) * 20
    dispute_score = (1 - get_dispute_rate(agent_id)) * 25
    uptime_score = get_uptime_rate(agent_id) * 15
    response_score = get_response_time_score(agent_id) * 10  # p99 vs SLA
    raw_score = base_attestation + txn_volume_score + dispute_score + uptime_score + response_score
    # Constraints
    if not is_kyb_verified(agent_id): raw_score = min(raw_score, 40)
    if has_active_dispute(agent_id): raw_score = min(raw_score, 50)
    if is_suspended(agent_id): return 0
    return int(raw_score)
```

### Sprint 2-C (Weeks 22-24): Tier 3 Financial Transactions + ERP Connectors

**Financial transactions** (requires external payment partner):

- Partner: Stripe Connect or Wise Business API for cross-border settlement
- Flow: ACME approves payment → HAR calls Stripe Connect to initiate wire → Baker Steel's Stripe account receives
- Escrow Contract: Fabric chaincode holds payment intent; releases on DELIVERY_CONFIRMED
- Constraint: financial transactions require KYB Level 3 (SWIFT BIC + bank account verified)

**ERP Connectors** (reconciliation agent v1):

- SAP S/4HANA: RFC/BAPI connector (create PO, GR posting, AP entry)
- Oracle NetSuite: SuiteScript REST connector
- Microsoft Dynamics 365: Dataverse API connector
- Generic Webhook: tenant-implemented; HAR posts reconciliation JSON

**Deliverables**:

- Full Hyperledger Fabric deployment with production chaincode
- KYB verification live (Stripe Identity)
- Trust score system computing accurately
- Tier 3 financial transactions (with pilot testing on small amounts)
- SAP and NetSuite reconciliation connectors

## Architecture Decisions

### AD-01: No Blockchain in Phase 1

**Decision**: Phase 1 uses a signed tamper-evident audit log (traditional database + signature chaining). Hyperledger Fabric deployed in Phase 2.

**Rationale**: Phase 1 validates concept. Blockchain adds 8+ weeks of infrastructure work and significant operational complexity. The tamper-evident log achieves 80% of blockchain's value for 20% of the cost. Phase 2 upgrades to true blockchain if traction is proven.

**Consequences**: Phase 1 audit log is controlled by mingai (not fully neutral). Clearly communicate this to Phase 1 participants: "Phase 1 uses a signed centralized log. Blockchain is coming in Phase 2."

---

### AD-02: HAR as Signing Proxy in Phase 1

**Decision**: In Phase 1, HAR generates and holds Ed25519 keys for all agents. HAR signs messages on agents' behalf.

**Rationale**: Requiring external developers to implement Ed25519 signing is a significant integration barrier for Phase 1. Centralizing signing reduces friction for the first cohort of participants.

**Consequences**: Phase 1 is cryptographically non-sovereign (HAR controls all keys). Phase 2 introduces BYOK (Bring Your Own Key) for participants who want self-sovereign agent identity.

---

### AD-03: Human Approval Gates Default to ON

**Decision**: All Tier 2 (PO placement) and Tier 3 (financial) transactions require human approval by default. Tenant admins can raise thresholds; cannot disable entirely for financial transactions.

**Rationale**: Liability protection. An AI agent committing a company to an unauthorized contract is a critical risk. Human approval gates are the primary safeguard.

**Consequences**: Some transaction automation benefit is lost. Acceptable tradeoff until legal frameworks for AI agent authority are established.

---

### AD-04: mingai-Hosted A2A Endpoint as Default

**Decision**: By default, mingai tenants use a HAR-hosted A2A endpoint. External messages route through HAR infrastructure to the tenant's workspace agent.

**Rationale**: Eliminates the requirement for tenants to expose a public HTTPS endpoint for their workspace agents. Most organizations cannot easily expose internal systems publicly.

**Consequences**: HAR is on the message path — not fully decentralized. All messages visible to HAR infrastructure. Privacy policy must disclose this. Phase 3 introduces self-hosted endpoint option.

## MVP Definition

**Minimum Viable Registry** (what must work at Week 6 for concept validation):

1. A tenant admin can publish an agent card from their workspace in < 15 minutes
2. A search query returns relevant agents in < 1 second
3. An A2A RFQ message can be sent from one registered agent to another
4. A response is returned and logged with a tamper-evident audit record
5. A human approval request is sent when PO threshold is exceeded

**What explicitly NOT in MVP**:

- Blockchain (Phase 2)
- KYB verification (Phase 2)
- Financial transactions (Phase 2)
- Third-party agents (Phase 3)
- ERP connectors (Phase 2)
- Mobile app (Phase 3+)
- Cross-registry federation (Phase 4+)
